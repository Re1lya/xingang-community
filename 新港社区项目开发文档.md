## 2. 项目概述

### 2.1 项目名称

项目中文名称：新港社区  
项目英文/仓库名称：xingang-community  
项目完整描述：新港社区：AI 本地生活服务与智能客服平台

### 2.2 项目定位

新港社区是一个面向本地生活场景的 AI 应用后端项目，为用户提供商户查询、优惠券秒杀、附近门店推荐、探店内容查询、智能客服和平台规则问答等能力。

项目基于 Spring Boot、Spring AI、MySQL、Redis、Redisson、MyBatis-Plus 等技术构建，在保留传统本地生活业务后端能力的基础上，引入大模型能力，实现以下AI应用特性：

1. 结构化规划：将用户自然语言请求解析为结构化执行计划。
2. Tool Calling：由模型调用Java业务工具查询真实业务数据。
3. RAG知识检索：检索平台规则、客服FAQ、优惠券说明和推荐约束。
4. Redis多轮会话记忆：保存最近对话、会话摘要和长期偏好事实。
5. SSE流式输出：面向智能客服和推荐问答提供增量响应体验。
6. Agent审计：记录traceId、用户、会话、意图、工具调用、知识命中和耗时。
7. Agent限流：基于Redis固定时间窗口控制AI接口访问频率。

### 2.3 核心业务目标

项目不是简单地在本地生活系统上增加一个聊天机器人，而是将本地生活业务系统进行 Agent 化改造。系统目标是让用户可以通过自然语言完成本地生活相关查询和决策，同时保证动态业务数据真实可信、推荐结果可解释、客服回答有规则依据、服务链路稳定可追踪。

核心业务目标包括：

1. 用户能够登录系统，并在登录态下进行商户查询、优惠券查询、秒杀下单和AI问答。
2. 用户能够查询商户详情、商户分类、附近门店和热门探店内容。
3. 用户能够参与优惠券秒杀，系统需要防止库存超卖和一人多单。
4. 用户能够通过自然语言提出推荐需求，例如地点、预算、人数、偏好、排除项等。
5. AI Agent能够将自然语言需求解析为结构化计划，并调用真实业务工具查询商户、优惠券、位置和推荐候选。
6. 智能客服能够基于RAG检索平台规则和客服知识，回答秒杀、优惠券、推荐限制、平台规则等问题。
7. 系统能够记录AI调用链路，支持问题排查、效果评估和后续风控分析。

## 4. 总体业务说明

### 4.1 用户登录业务

用户登录用于建立用户身份上下文。登录成功后，系统将用户身份与token写入Redis。后续接口通过token识别当前用户，AI Agent可以基于当前用户读取会话记忆、位置上下文、长期偏好事实和限流配额。

核心要求：

1. 用户登录成功后必须生成唯一token。
2. token必须写入Redis，并设置合理过期时间。
3. 请求进入业务接口前必须解析当前用户。
4. AI接口审计必须记录userId或匿名principalKey。
5. 未登录或token失效时，业务接口应返回统一鉴权错误。

当前原始文档落点：

| 能力 | 当前落点 |
| --- | --- |
| 用户登录 | `UserController` |
| 登录态保存 | Redis token |
| 用户上下文 | 登录态、鉴权、会话用户上下文 |

### 4.2 商户查询业务

商户查询是本地生活业务底座，提供商户详情、商户分类、按分类查询、按位置查询、附近门店查询等能力。

核心要求：

1. 商户详情优先查询Redis缓存，缓存未命中再查询MySQL。
2. 非法商户ID导致数据库无数据时，应缓存空值，防止缓存穿透。
3. 热点商户缓存失效时，应使用互斥锁或逻辑过期策略降低缓存击穿风险。
4. 商户分类应缓存到Redis，减少数据库查询。
5. 附近门店应使用Redis GEO能力，按分类、经纬度、距离和分页条件查询。
6. 商户数据更新后必须删除或刷新相关缓存，避免脏读。

当前原始文档落点：

| 能力 | 当前落点 |
| --- | --- |
| 商户查询 | `ShopController`, `ShopServiceImpl` |
| 商户缓存 | `CacheClient`, `RedisConstants` |
| 商户GEO | `SHOP_GEO_KEY`, Redis GEO查询 |

### 4.3 优惠券秒杀业务

优惠券秒杀用于处理高并发抢券场景。系统必须保证库存不超卖、用户不能重复下单、接口响应尽量快速、数据库不会被瞬时高并发压垮。

核心链路：

```text
用户抢券
  ↓
生成订单ID
  ↓
执行 lua/seckill.lua
  ↓
校验库存是否充足
  ↓
校验用户是否已下单
  ↓
Redis预扣库存
  ↓
SADD记录用户已抢券
  ↓
XADD写入Redis Stream
  ↓
后台消费组异步读取订单
  ↓
Redisson用户锁防重复处理
  ↓
事务内二次校验 + MySQL乐观扣减库存 + 保存订单
  ↓
ACK消息
```

核心要求：

1. 秒杀入口线程只负责资格判断、库存预扣和消息入队，不直接写入最终订单。
2. Lua脚本必须将库存判断、一人一单判断、扣减库存、记录用户购买集合、写入Redis Stream合并为一次原子操作。
3. Redis Stream消费者必须使用消费组读取订单消息。
4. 消费异常时必须处理pending-list，降低消息丢失风险。
5. 订单创建成功后必须ACK消息。
6. 异步创建订单前必须按用户维度加Redisson分布式锁。
7. 事务内必须二次校验一人一单和数据库库存。
8. MySQL扣减库存必须使用乐观扣减条件，防止最终库存超卖。

关键Redis Key：

| Key | 用途 |
| --- | --- |
| `seckill:stock:{voucherId}` | 秒杀券Redis预扣库存。 |
| `seckill:order:{voucherId}` | 已抢券用户集合，用于一人一单校验。 |
| `stream.orders` | 秒杀订单异步消息流。 |
| `lock:order:{userId}` | 用户维度分布式锁。 |

当前原始文档落点：

| 能力 | 当前落点 |
| --- | --- |
| 优惠券秒杀 | `VoucherOrderServiceImpl`, `lua/seckill.lua` |
| 异步下单 | Redis Stream `stream.orders` |
| 分布式锁 | Redisson `lock:order:{userId}` |

### 4.4 AI Agent业务

AI Agent用于承接用户自然语言请求，并根据请求类型完成推荐、客服、事实查询等任务。

Agent核心边界如下：

```text
动态事实：Tool / DB / Redis
静态知识：RAG
表达生成：LLM
链路追踪：toolTrace + retrievalHits + audit stream
```

动态事实包括：

1. 门店价格。
2. 门店评分。
3. 门店距离。
4. 优惠券库存。
5. 当前优惠信息。
6. 营业状态。
7. 附近门店列表。
8. 用户当前会话上下文。

动态事实必须通过Tool、MySQL或Redis查询，不得由RAG或大模型编造。

静态知识包括：

1. 平台规则。
2. 客服FAQ。
3. 优惠券说明。
4. 秒杀规则说明。
5. 推荐约束说明。
6. 平台无法保证事项说明。

静态知识应通过RAG检索补充，不应用作实时库存、实时距离、实时价格等动态事实来源。

当前原始文档落点：

| 能力 | 当前落点 |
| --- | --- |
| Agent API | `AiAgentController` |
| 结构化规划 | `AgentPlanningServiceImpl`, `AgentExecutionPlan` |
| Tool Calling | `LocalLifeAgentTools` |
| RAG | `LocalLifeRagService`, `AiKnowledgeServiceImpl` |
| 推荐服务 | `ShopRecommendationServiceImpl` |
| 会话记忆 | `RedisChatMemoryRepository`, `RedisChatContextRepository` |
| 上下文压缩 | `ContextCompressionService` |
| Agent限流 | `AgentRateLimitServiceImpl` |
| Agent审计 | `AgentAuditServiceImpl` |

### 4.5 智能客服RAG业务

智能客服RAG用于回答平台规则、优惠券说明、秒杀说明、推荐规则和客服常见问题。

知识库入口：

```text
src/main/resources/knowledge/zyro-support.json
```

当前配置能力：

| 配置项 | 用途 |
| --- | --- |
| `AI_RAG_ENABLED` | 控制RAG能力是否开启。 |
| `AI_RAG_STORE_FILE` | 指定知识索引或知识存储文件。 |
| `AI_RAG_SIMILARITY_THRESHOLD` | 控制知识召回相似度阈值。 |
| `AI_KNOWLEDGE_TOP_K` | 控制召回知识条数。 |
| `AI_EMBEDDING_PROVIDER` | 控制Embedding提供方。 |
| `AI_RERANK_ENABLED` | 控制是否开启重排。 |

适合RAG回答的问题：

1. 优惠券过期了还能用吗？
2. 秒杀券抢到了在哪里看？
3. 为什么推荐结果里没有某家店？
4. 商家评分和评价有什么参考意义？
5. 平台为什么不能保证历史价格和库存？

不适合RAG直接回答的问题：

1. 这张券现在还有库存吗？
2. 这家店现在有什么优惠？
3. 我附近3公里有哪些店？
4. 这个商户今天营业吗？

上述不适合RAG直接回答的问题必须走Tool查询动态事实。

### 4.6 门店推荐业务

门店推荐Agent用于处理用户自然语言推荐需求。用户可能提供城市、地点、预算、人数、场景、偏好、排除项、距离范围等条件。

示例用户输入：

```text
广州正佳附近有没有适合两个人约会、人均80以内、不要火锅的餐厅？
```

Planner预期结构化结果：

```json
{
  "intent": "recommendation",
  "city": "广州",
  "locationHint": "正佳",
  "nearby": true,
  "partySize": 2,
  "budgetMax": 80,
  "scenePreference": "约会",
  "excludedCategories": ["火锅"],
  "preferredTools": ["recommend_nearby_shops", "get_shop_coupons"]
}
```

推荐执行链路：

```text
Planner解析结构化计划
  ↓
Redis读取会话上下文和长期偏好事实
  ↓
百度地图解析显式地点坐标
  ↓
推荐服务查询DB / GEO / 语义候选
  ↓
按预算、位置、优惠、排除项排序
  ↓
Tool返回真实候选
  ↓
RAG只补充推荐规则或平台解释
  ↓
LLM生成最终推荐回答
```

推荐输出必须包含：

1. 门店名称。
2. 人均价格。
3. 评分。
4. 距离或商圈。
5. 优惠券情况。
6. 推荐理由。
7. 无法保证的条件说明。

推荐输出禁止包含：

1. Tool未返回的虚构门店。
2. 未查询到的虚构优惠券。
3. 无依据的营业状态承诺。
4. 无依据的库存承诺。
5. 无依据的距离和价格。

### 4.7 多轮会话记忆业务

AI Agent需要支持多轮追问。系统不应只保存最近N轮原文对话，还应保存会话摘要、长期偏好事实，并通过上下文压缩控制token成本。

当前会话记忆能力包括：

1. 最近对话窗口。
2. 会话摘要。
3. 长期偏好事实。
4. MicroCompact压缩。
5. token budget控制。
6. 记忆过期。
7. 相关性过滤。

多轮示例：

```text
第一轮：帮我推荐正佳附近适合约会的餐厅，人均80以内
第二轮：不要西餐，换成粤菜
第三轮：有没有券？
```

预期行为：

1. 第二轮继承地点“正佳”、场景“约会”、预算“人均80以内”。
2. 第二轮新增排除“西餐”和偏好“粤菜”。
3. 第三轮基于上一轮候选或相同约束查询优惠券。
4. 若上下文缺失或候选为空，应明确说明无法确认，不得编造。

### 4.8 SSE流式响应业务

AI Agent提供同步响应和SSE流式响应。流式响应用于提升用户等待体验，适合推荐理由生成、客服问答和长文本解释。

当前接口：

```text
POST /ai/agent/chat/stream
```

当前事件：

| 事件 | 含义 |
| --- | --- |
| `meta` | 返回会话元信息、traceId、规划摘要等。 |
| `chunk` | 返回模型生成的增量内容。 |
| `done` | 返回最终完成事件。 |
| `error` | 返回统一错误事件。 |

后续可增强事件：

| 事件 | 含义 |
| --- | --- |
| `stage: planning` | 正在进行结构化规划。 |
| `stage: tool_call` | 正在调用业务工具。 |
| `stage: rag` | 正在检索知识库。 |
| `stage: generating` | 正在生成最终回答。 |

后续增强事件只作为迭代方向，当前不得写成已完成能力。

### 4.9 Agent审计与限流业务

Agent接口需要具备限流和审计能力。

当前限流实现：

```text
Redis固定时间窗口计数
chat和stream分别配置请求上限
key = ai:rate-limit:{scene}:{principalKey}:{bucket}
```

限流要求：

1. 普通对话和流式对话必须区分限流场景。
2. 已登录用户优先使用userId作为principalKey。
3. 未登录或匿名请求使用IP、设备标识或匿名key作为principalKey。
4. 超过限流阈值时返回统一限流错误。
5. 限流错误不得继续调用大模型，避免成本浪费。

当前审计实现：

```text
异步写入Redis Stream
默认stream key = hmdp:agent:audit
记录traceId、userId、conversationId、intent、toolTrace、retrievalCount、latencyMs
```

审计要求：

1. 每次Agent请求必须生成或透传traceId。
2. 审计记录必须包含用户、会话、意图、知识召回数量、工具轨迹和耗时。
3. 审计写入失败不得影响主流程回答，但必须记录日志。
4. 审计字段中不得写入用户敏感隐私原文，必要时应脱敏或摘要化。

## 5. 技术架构

### 5.1 总体架构

```text
用户请求
  ↓
API层：登录鉴权 / traceId / HTTP / SSE
  ↓
业务层：商户查询 / 优惠券秒杀 / 探店博客 / 推荐服务
  ↓
缓存与并发层：Redis缓存 / Redis GEO / Lua / Redis Stream / Redisson
  ↓
Agent层：Planner结构化规划 / Tool Calling / Redis会话记忆
  ↓
RAG层：平台规则 / 客服FAQ / 优惠券说明 / 推荐约束
  ↓
LLM层：基于工具结果和知识片段生成自然语言回答
  ↓
治理层：限流 / 审计 / trace / Actuator / Prometheus
```

### 5.2 分层说明

| 层级 | 职责 | 典型组件 |
| --- | --- | --- |
| API层 | 接收HTTP请求、解析参数、返回统一响应、提供SSE流式输出。 | Controller、Request DTO、Response VO。 |
| 鉴权上下文层 | 解析token、维护当前用户上下文、生成traceId。 | UserHolder、登录拦截器、Trace过滤器。 |
| 业务服务层 | 承载商户、优惠券、订单、推荐等业务逻辑。 | Service、ServiceImpl。 |
| 数据访问层 | 访问MySQL，执行CRUD和分页查询。 | Mapper、Entity、MyBatis-Plus。 |
| 缓存并发层 | 提供缓存、GEO、Lua、Stream、锁等能力。 | RedisTemplate、StringRedisTemplate、Redisson、Lua脚本。 |
| AI编排层 | 执行结构化规划、工具选择、结果整合。 | AgentPlanningService、LocalLifeAgentTools。 |
| RAG层 | 负责知识加载、向量检索、重排和知识命中。 | LocalLifeRagService、AiKnowledgeService。 |
| 模型层 | 调用OpenAI-compatible Chat Model生成最终回答。 | Spring AI Chat Model。 |
| 治理层 | 限流、审计、监控、健康检查。 | AgentRateLimitService、AgentAuditService、Actuator。 |

### 5.3 AI链路架构

AI链路必须遵循“先规划、再查事实、再检索规则、最后生成表达”的顺序。

```text
用户自然语言输入
  ↓
身份识别与限流
  ↓
读取Redis会话记忆和长期偏好事实
  ↓
Planner生成AgentExecutionPlan
  ↓
判断意图：推荐 / 客服 / 事实查询 / 其他
  ↓
根据意图调用Tool查询动态事实
  ↓
需要规则解释时调用RAG检索静态知识
  ↓
整合Tool结果、RAG片段和会话上下文
  ↓
调用LLM生成最终回答
  ↓
写入会话记忆、审计Stream和响应结果
```

### 5.4 秒杀链路架构

秒杀链路必须遵循“Redis前置校验、Stream削峰、DB最终确认”的顺序。

```text
请求进入秒杀接口
  ↓
校验登录态和voucherId
  ↓
生成全局订单ID
  ↓
执行Lua脚本
  ↓
Lua返回结果：
    0 = 成功入队
    1 = 库存不足
    2 = 重复下单
  ↓
接口快速返回订单ID或错误
  ↓
后台消费者读取stream.orders
  ↓
加Redisson用户锁
  ↓
事务内检查订单是否存在
  ↓
乐观扣减MySQL库存
  ↓
保存订单
  ↓
ACK消息
```

## 7. 全局公共模块说明

本章内容属于产品经理负责的全局通用模块、公共基础类、公共工具和全局配置。所有业务模块均依赖本章能力，不得私自复制、绕过或随意改动。

### 7.1 统一返回结构

所有HTTP接口必须返回统一结构。推荐字段如下：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `success` | Boolean | 是 | 请求是否成功。 |
| `code` | String / Integer | 是 | 业务状态码。 |
| `message` | String | 是 | 用户可理解的提示信息。 |
| `data` | Object | 否 | 业务数据。 |
| `traceId` | String | 是 | 链路追踪ID。 |
| `timestamp` | Long | 是 | 响应时间戳。 |

成功示例：

```json
{
  "success": true,
  "code": "0",
  "message": "success",
  "data": {
    "id": 1
  },
  "traceId": "trace-20260508-0001",
  "timestamp": 1778236800000
}
```

失败示例：

```json
{
  "success": false,
  "code": "SECKILL_STOCK_NOT_ENOUGH",
  "message": "优惠券库存不足",
  "data": null,
  "traceId": "trace-20260508-0002",
  "timestamp": 1778236800000
}
```

### 7.2 统一异常处理

必须建立全局异常处理器。异常分类如下：

| 异常类型 | 场景 | 处理方式 |
| --- | --- | --- |
| 参数异常 | 入参为空、格式错误、范围不合法。 | 返回参数错误码和明确字段提示。 |
| 鉴权异常 | token缺失、token过期、用户未登录。 | 返回未登录或无权限错误。 |
| 业务异常 | 库存不足、重复下单、商户不存在。 | 返回业务错误码。 |
| 限流异常 | AI接口超过限流。 | 返回限流错误，不调用模型。 |
| 外部服务异常 | 模型、地图、Redis、MySQL异常。 | 记录日志，按场景降级或返回系统繁忙。 |
| 系统异常 | 未捕获运行时异常。 | 返回统一系统错误，隐藏内部堆栈。 |

异常响应必须包含traceId，方便开发人员和维护人员定位日志。

### 7.3 统一traceId

每个请求必须具备traceId。规则如下：

1. 如果请求头传入traceId，则透传。
2. 如果请求头未传入traceId，则服务端生成。
3. traceId必须写入日志MDC。
4. AI审计、接口响应、异常响应、外部调用日志均必须携带traceId。

### 7.4 统一用户上下文

用户上下文用于在业务层获取当前登录用户。规则如下：

1. Controller不得自行解析Redis token，应通过统一拦截器或上下文工具获取当前用户。
2. Service层需要userId时，从统一上下文读取。
3. 异步线程使用用户上下文时，必须显式传递必要字段，不得依赖ThreadLocal自动继承。
4. AI审计必须记录userId；无登录态时记录匿名principalKey。

### 7.5 公共Redis封装

公共Redis封装由产品经理维护。业务模块不得重复编写Redis序列化、锁、缓存空值、逻辑过期等基础代码。

公共Redis封装至少包含：

1. 普通字符串读写。
2. JSON对象序列化与反序列化。
3. 缓存空值写入。
4. 互斥锁获取与释放。
5. 逻辑过期数据封装。
6. Redis GEO写入与查询。
7. Redis Stream写入与读取辅助。
8. 固定窗口限流计数。

### 7.6 公共AI封装

公共AI封装用于统一模型调用、提示词模板、Tool注册、RAG配置和输出解析。

要求如下：

1. 模型调用配置不得散落在业务代码中。
2. Prompt模板应集中管理，便于审核和调优。
3. Tool返回结构必须稳定，禁止返回不可解析的大段自由文本。
4. RAG检索阈值、TopK、重排开关必须可配置。
5. AI异常必须可降级，不能导致整个业务服务不可用。

## 10. Redis设计

### 10.1 Redis用途总览

| 用途 | 说明 |
| --- | --- |
| 登录token | 保存用户登录态。 |
| 商户缓存 | 缓存商户详情和商户分类。 |
| 缓存空值 | 防止非法ID穿透数据库。 |
| 互斥锁 | 防止热点Key缓存击穿。 |
| 逻辑过期 | 降低热点商户重建缓存压力。 |
| Redis GEO | 支持附近门店查询。 |
| 秒杀库存 | Redis预扣库存。 |
| 一人一单集合 | 记录已抢券用户。 |
| Redis Stream | 秒杀订单异步队列、Agent审计流。 |
| Redisson锁 | 用户维度分布式锁。 |
| Agent限流 | 固定时间窗口计数。 |
| Agent会话记忆 | 最近对话、会话摘要、偏好事实。 |

### 10.2 关键Key规范

| Key | 类型 | 说明 | 负责人 |
| --- | --- | --- | --- |
| `login:token:{token}` | String / Hash | 用户登录态。 | 产品经理 |
| `cache:shop:{id}` | String | 商户详情缓存。 | 开发人员A |
| `cache:shop:type` | String / List | 商户分类缓存。 | 开发人员A |
| `lock:shop:{id}` | String | 商户缓存重建互斥锁。 | 开发人员A |
| `shop:geo:{typeId}` 或 `SHOP_GEO_KEY` | GEO | 分类维度商户坐标。 | 开发人员A |
| `seckill:stock:{voucherId}` | String | 秒杀券库存。 | 开发人员A |
| `seckill:order:{voucherId}` | Set | 已抢券用户集合。 | 开发人员A |
| `stream.orders` | Stream | 秒杀订单消息。 | 开发人员A |
| `lock:order:{userId}` | Redisson Lock | 用户维度订单锁。 | 开发人员A |
| `ai:rate-limit:{scene}:{principalKey}:{bucket}` | String | AI限流计数。 | 开发人员B |
| `hmdp:agent:audit` | Stream | Agent审计流。 | 开发人员B |
| `ai:chat:memory:{conversationId}` | String / List | 最近对话窗口。 | 开发人员B |
| `ai:chat:context:{conversationId}` | String | 会话摘要和上下文。 | 开发人员B |
| `ai:user:fact:{userId}` | String / Set | 长期偏好事实。 | 开发人员B |

### 10.3 Redis Key管理规则

1. 所有Key前缀必须集中定义在常量类中。
2. 不允许在业务代码中硬编码散落Key字符串。
3. Key中变量必须使用明确含义，例如 `{userId}`、`{voucherId}`、`{conversationId}`。
4. 缓存Key必须设置TTL；特殊长期数据必须说明原因。
5. 锁Key必须设置过期时间，防止死锁。
6. Stream消费成功后必须ACK。
7. 审计Stream需要制定清理策略，避免无限增长。

### 12.1 商户缓存模块

负责人：开发人员A  
审核人：产品经理

模块目标：

提升商户详情和商户分类等高频读接口性能，降低数据库压力，防止缓存穿透和缓存击穿。

核心能力：

1. 商户详情缓存。
2. 商户分类缓存。
3. 缓存空值防穿透。
4. 互斥锁防击穿。
5. 逻辑过期防击穿。
6. 数据库更新后删除缓存。

开发要求：

1. 查询商户详情时必须优先查询缓存。
2. 查询到空值缓存时直接返回商户不存在。
3. 查询数据库无数据时写入短TTL空值缓存。
4. 使用互斥锁重建缓存时，锁必须设置过期时间。
5. 使用逻辑过期时，旧数据可短暂返回，但必须异步重建缓存。

验收标准：

1. 商户详情接口可正常返回数据。
2. 非法ID连续请求不会反复打到数据库。
3. 热点商户缓存过期时不会大量线程同时重建缓存。
4. 商户更新后缓存不会长期返回旧值。

### 12.3 秒杀优惠券模块

负责人：开发人员A

验收标准：

1. 正常抢券返回订单ID。

## 13. 编码规范

### 13.1 命名规范

1. 类名使用大驼峰，例如 `ShopServiceImpl`。
2. 方法名使用小驼峰，例如 `queryShopById`。
3. 常量名使用全大写下划线，例如 `CACHE_SHOP_KEY`。
4. DTO类以 `DTO` 结尾。
5. VO类以 `VO` 结尾。
6. Entity类与数据库业务对象一致。
7. Controller方法名称必须体现业务含义，不得使用 `test`、`doIt`、`handle` 等模糊命名。

### 13.2 分层规范

1. Controller只做参数接收、参数校验和响应返回。
2. Service承载业务逻辑。
3. Mapper只做数据访问。
4. Tool类只做AI工具适配，不直接承载复杂业务。
5. 公共工具放入common或infrastructure，不得放入业务模块内部复用。

### 13.3 注释规范

必须添加注释的场景：

1. 类的职责说明。
2. 对外暴露的Service方法。
3. 复杂业务流程。
4. Lua脚本调用参数和返回值。
5. Redis Key用途。
6. AI Prompt关键约束。
7. RAG和Tool边界。
8. 分布式锁和事务边界。

不推荐的注释：

1. 重复代码本身含义的注释。
2. 过期注释。
3. 与实际逻辑不一致的注释。
4. 无意义的作者标记。

### 13.4 日志规范

必须记录日志的场景：

1. 登录失败。
2. 秒杀Lua返回异常。
3. Redis Stream消费失败。
4. pending-list处理失败。
5. Redisson加锁失败。
6. AI模型调用失败。
7. RAG检索失败。
8. Tool调用失败。
9. Agent审计写入失败。
10. 外部地图服务调用失败。

日志必须包含：

1. traceId。
2. userId或principalKey。
3. 关键业务ID，例如shopId、voucherId、orderId、conversationId。
4. 错误码。
5. 异常摘要。

日志不得包含：

1. 密码。
2. token完整值。
3. API Key。
4. 用户敏感隐私。
5. 大段模型输入输出原文，除非经过脱敏和采样。

### 13.5 事务规范

1. 数据库写操作必须明确事务边界。
2. 秒杀订单创建必须使用事务。
3. 事务方法应由Spring代理对象调用，避免事务失效。
4. 事务中不得执行耗时外部模型调用。
5. 事务中尽量避免长时间持有分布式锁。

### 13.6 Redis规范

1. Redis Key必须集中定义。
2. 缓存必须设置TTL。
3. 锁必须设置过期时间。
4. 使用Lua时必须明确KEYS和ARGV。
5. Redis Stream消费成功必须ACK。
6. Redis异常必须有降级或明确错误处理。

### 13.7 AI输出规范

1. 模型不得编造Tool未返回的商户。
2. 模型不得编造优惠券、库存、距离、营业状态。
3. 模型回答必须明确哪些信息来自实时查询，哪些信息来自平台规则。
4. RAG无命中时不得假装知道规则。
5. 推荐结果为空时必须说明为空原因和可调整条件。
6. 涉及退款、赔付、投诉等高风险场景时，应低置信度转人工或明确无法确认。

1. 开发人员完成模块开发。

5. 产品经理按本文档标准进行Code Review。

7. 修改完成后再次提交审核。
8. 审核通过后允许合并。

### 14.2 通用审核项

| 审核项 | 要求 |
| --- | --- |
| 需求一致性 | 代码必须实现指定需求，不得私自扩张范围。 |
| 模块边界 | 不得跨模块直接修改他人负责逻辑。 |
| 分层清晰 | Controller、Service、Mapper职责清晰。 |
| 命名规范 | 类名、方法名、字段名清晰准确。 |
| 异常处理 | 必须返回统一错误，不得直接抛出内部异常给用户。 |

| 参数校验 | 入参必须校验必填、格式、范围。 |
| 安全 | 不得泄露token、密码、API Key。 |
| 性能 | 不得出现明显N+1查询、全表扫描、无界循环。 |
| 可维护性 | 不得复制大量重复代码。 |

### 14.3 秒杀模块审核项

1. Lua脚本是否保证库存判断和扣减原子性。
2. 是否校验一人一单。
3. 是否写入Redis Stream。
4. Redis Stream消费者是否ACK。
5. 是否处理pending-list。
6. 是否使用Redisson用户锁。
7. 数据库是否二次校验订单。
8. 数据库库存扣减是否带 `stock > 0` 条件。
9. 异常情况下是否会导致库存和订单不一致。

### 14.4 AI模块审核项

1. Planner输出是否结构化。
2. Tool入参是否校验。
3. Tool返回是否结构化。
4. 动态事实是否全部来自Tool或数据库。
5. RAG是否只用于静态规则。
6. 模型是否可能编造商户、券、库存或距离。
7. 多轮上下文是否继承正确。
8. SSE是否包含meta、chunk、done、error。
9. 限流是否在模型调用前执行。
10. 审计是否记录traceId、intent、toolTrace、retrievalCount、latencyMs。

### 14.5 拒绝合并条件

出现以下任一情况，代码不得合并：

1. 未通过本地自测。
2. 存在编译错误。
3. 接口返回结构不统一。
4. 绕过公共封装私自访问Redis或模型配置。
5. AI回答可能编造动态事实。
6. 秒杀链路可能导致超卖或重复下单。
7. 泄露敏感信息。
8. 未处理核心异常。
9. 未写必要注释。
10. 修改了公共模块但未经过产品经理确认。

## 18. 异常处理与降级策略

### 18.1 Redis异常

影响：

1. 登录态可能不可用。
2. 商户缓存不可用。
3. 秒杀链路不可用。
4. Agent限流、记忆、审计不可用。

处理策略：

1. 商户普通查询可降级到MySQL，但必须注意数据库压力。
2. 秒杀链路依赖Redis Lua和Stream，Redis异常时应直接返回系统繁忙，不得绕过Redis直接下单。
3. Agent记忆异常时可降级为单轮对话。
4. Agent审计异常时记录日志，不影响主回答。
5. 限流异常时应采用保守策略，避免无限调用模型。

### 18.2 MySQL异常

影响：

1. 商户、优惠券、订单等业务数据不可用。
2. 秒杀最终订单无法创建。

处理策略：

1. 读接口可在缓存命中时继续返回缓存数据。
2. 写接口应返回系统繁忙。
3. 秒杀消费者创建订单失败时不得ACK消息，应进入重试或pending-list处理。

### 18.3 AI模型异常

影响：

1. Agent回答生成失败。
2. Planner可能不可用。

处理策略：

1. 返回统一AI服务不可用错误。
2. 如果Tool查询已经成功，可返回结构化候选摘要作为降级结果。
3. 不得返回空白响应。
4. 必须写入日志和审计失败信息。

### 18.4 RAG无命中或低置信度

处理策略：

1. 不得编造平台规则。
2. 如果问题属于动态事实，改走Tool。
3. 如果问题属于平台规则但无命中，应提示暂无法基于现有规则确认。
4. 后续增强可记录到未解决问题表。

推荐用户侧回答：

```text
这个问题我暂时无法基于现有平台规则确认。为了避免误导你，建议转人工客服，我也会把这个问题记录下来用于后续补充知识库。
```

### 18.5 百度地图异常

处理策略：

1. 如果用户提供经纬度，则优先使用用户经纬度。
2. 如果只有地点文本且地图解析失败，应提示用户补充更明确位置。
3. 不得编造经纬度或距离。
