# 新港社区前端开发文档

文档版本：V1.0  
编制日期：2026-05-12  
适用项目：新港社区：AI 本地生活服务与智能客服平台  
适用范围：React 前端工程规划、页面设计、接口对接、交互规范、后续开发验收

---

## 1. 文档说明

本文档用于指导“新港社区”前端工程的后续开发。当前仓库已经具备后端、数据库、Redis、秒杀、AI Agent、RAG、SSE、限流和审计等基础能力，但尚未创建正式前端工程。

第一轮已创建 `frontend/` React + TypeScript + Vite 工程骨架，并完成前端骨架 Review 修复。当前仍不修改后端接口、不新增大型 UI 框架、不伪造动态事实。

后续如果正式开发前端，应以本文档和 `新港社区项目开发文档.md` 为共同基准。

当前保留 `PRODUCT.md` 作为 `$impeccable` 前端产品上下文文件，用于记录产品定位、设计原则、反例和可访问性边界，帮助后续 AI 在改 UI 时保持一致。

---

## 2. 前端项目定位

前端项目定位为“类似美团风格的本地生活服务与 AI 智能推荐前端”。

它面向三类用户：

1. 普通用户：登录、浏览商户、查看优惠券、参与秒杀、咨询 AI 客服。
2. 演示人员：快速展示本地生活业务链路、秒杀链路和 AI Agent 能力。
3. 开发和测试人员：验证后端接口、SSE 流式响应、错误处理和 traceId 链路。

前端不直接访问数据库，不绕过后端 Service，不伪造动态事实。商户价格、距离、优惠券、库存、营业状态、AI 工具轨迹等数据必须来自后端接口。

---

## 3. 技术栈规划

建议前端技术栈如下：

| 技术 | 作用 |
| --- | --- |
| React | 页面组件化开发，负责 UI 渲染和交互。 |
| TypeScript | 为接口数据、组件属性、状态和事件增加类型检查，降低多人协作错误。 |
| Vite | 前端工程构建工具，负责开发服务器、热更新、打包和环境变量。 |
| React Router | 前端路由管理，负责页面跳转和 URL 映射。 |
| TanStack Query | 服务端数据状态管理，负责接口请求、缓存、加载态、错误态和重试策略。 |
| Zustand | 轻量全局状态管理，负责 token、用户信息、定位、AI 会话等前端状态。 |
| Axios | 普通 HTTP 请求封装，负责 baseURL、请求头、token、响应拦截和错误处理。 |
| fetch ReadableStream / SSE | 对接 AI 流式响应。当前后端流式接口为 POST，优先使用 fetch stream；仅在后端新增 GET SSE 时使用 EventSource。 |

说明：React 本身可以使用 JavaScript 开发，但本项目后端接口结构较多，AI 响应结构也较复杂，推荐使用 TypeScript 作为默认开发语言。

## 4. 视觉风格定位

前端整体参考美团本地生活产品的交互气质，但不得使用美团商标、品牌素材或受版权保护的图片资源。

### 4.1 设计关键词

1. 本地生活。
2. 清晰实用。
3. 黄色主视觉。
4. 卡片式商户列表。
5. 优惠券和秒杀强提示。
6. 移动端优先。
7. AI 客服入口轻量、可随时唤起。

### 4.2 推荐色彩

| 用途 | 建议颜色 |
| --- | --- |
| 主色 | `#FFD100` 或接近的暖黄色 |
| 强调色 | `#FF6A00`，用于秒杀、优惠券、限时提示 |
| 成功色 | `#00A870` |
| 警告色 | `#FFB020` |
| 错误色 | `#D93026` |
| 主文字 | `#222222` |
| 次文字 | `#666666` |
| 辅助文字 | `#999999` |
| 页面背景 | `#F5F5F5` |
| 卡片背景 | `#FFFFFF` |

### 4.3 UI 规范

1. 页面以移动端宽度优先设计，再适配桌面端。
2. 商户、优惠券、AI 推荐结果使用卡片承载。
3. 卡片圆角不超过 8px，避免过度圆润。
4. 秒杀按钮必须根据状态显示不同文案：未开始、立即抢、已抢过、已抢光、已结束、处理中。
5. 优惠券金额必须明确单位，后端金额字段以“分”为主，前端展示时转换为“元”。
6. AI 对话页必须区分用户消息、AI 消息、工具调用摘要和错误提示。
7. 不使用大面积渐变背景，不做营销落地页式首屏，首页应直接呈现可操作的本地生活功能。

## 5. 页面与路由规划

| 路由 | 页面 | 主要职责 |
| --- | --- | --- |
| `/login` | 登录页 | 输入手机号，调用登录接口，保存 token 和用户信息。 |
| `/` | 首页 | 搜索框、分类入口、附近商户、秒杀入口、AI 客服入口。 |
| `/shops` | 商户列表页 | 按分类、地点、分页、经纬度查询商户。 |
| `/shops/:shopId` | 商户详情页 | 展示商户详情、优惠券列表、秒杀券入口。 |
| `/vouchers/:shopId` | 商户优惠券页 | 展示某商户全部优惠券和秒杀券。 |
| `/seckill/result/:orderId` | 秒杀结果页 | 展示抢券排队成功、处理中、失败原因和后续提示。 |
| `/ai/chat` | AI 对话页 | 普通问答、推荐问答、客服问答、SSE 流式回答。 |
| `/ai/session` | AI 会话管理页 | 清理当前会话，展示 conversationId 和 traceId。 |
| `/error` | 错误页 | 统一展示异常、traceId 和重试入口。 |

后续如需管理端页面，例如知识库重建、秒杀库存重置、Stream 监控，应独立规划为管理后台，不混入普通用户端。

## 6. 核心页面说明

### 6.1 登录页

业务目标：让用户快速进入系统，拿到后端 token。

接口：

```text
POST /user/login
```

请求体：

```json
{
  "phone": "13800000001",
  "code": "",
  "password": ""
}
```

前端处理：

1. 校验手机号不能为空。
2. 调用登录接口。
3. 保存 `token`、`userId`、`nickName`。
4. 跳转首页。

### 6.2 首页

业务目标：作为本地生活主入口。

模块建议：

1. 顶部定位和搜索。
2. 分类入口，例如美食、休闲娱乐。
3. 推荐商户卡片。
4. 限时秒杀区域。
5. AI 智能推荐入口。

首页不应做纯宣传页，首屏必须能直接进入业务操作。

当前首页商户区域已经改为调用 `GET /shop/of/type` 查询真实商户。若后端暂无数据，应展示空状态或错误状态，不得用未标注的假商户充当真实结果。

### 6.3 商户列表页

接口：

```text
GET /shop/of/type?typeId=1&current=1&pageSize=10
GET /shop/of/type?typeId=1&current=1&pageSize=10&x=116.3103&y=39.9836
```

前端处理：

1. 有定位经纬度时启用附近查询。
2. 无定位时按分类分页查询。
3. 显示商户名称、区域、地址、人均价格、评分、距离。
4. 人均价格从“分”转换为“元”展示。
5. 当前后端未提供 `/shop/type` 分类接口，前端先使用文档明确的演示分类常量，例如美食、咖啡、休闲，并在页面标注“演示分类”。后续如后端补分类接口，再替换为真实分类查询。
6. 后端 `score` 字段按实际评分展示，例如 `4.6` 显示为 `4.6 分`，前端不得再除以 10。

### 6.4 商户详情页

接口：

```text
GET /shop/{shopId}
GET /voucher/list/{shopId}
```

前端处理：

1. 并行查询商户详情和优惠券列表。
2. 普通券展示规则、面值、使用说明。
3. 秒杀券展示库存、开始时间、结束时间和抢购按钮。
4. 商户不存在时展示空状态，不跳白屏。

### 6.5 秒杀交互

接口：

```text
POST /voucher-order/seckill/{voucherId}
```

前端状态：

| 后端结果 | 前端展示 |
| --- | --- |
| 成功返回 orderId | 显示“已进入抢券队列”，跳转秒杀结果页。 |
| `SECKILL_STOCK_NOT_ENOUGH` | 显示“已抢光”。 |
| `SECKILL_DUPLICATE_ORDER` | 显示“你已经抢过该券”。 |
| `SECKILL_NOT_STARTED` | 显示“活动未开始”。 |
| `SECKILL_ENDED` | 显示“活动已结束”。 |
| `UNAUTHORIZED` | 跳转登录页。 |
| 其他错误 | 显示失败提示和 traceId。 |

交互要求：

1. 点击按钮后立即进入 loading 状态，防止重复点击。
2. 不在前端自行扣库存，只展示后端返回结果。
3. 成功返回订单 ID 只代表入队成功，不代表 MySQL 最终订单已经落库。
4. 后续如后端提供订单查询接口，再补轮询或结果查询。
5. 秒杀接口失败时必须在当前页面展示后端 `message` 和 `traceId`，不得只跳转成功页或吞掉错误。

### 6.6 AI 对话页

普通问答接口：

```text
POST /ai/agent/chat
```

流式问答接口：

```text
POST /ai/agent/chat/stream
```

请求体：

```json
{
  "message": "附近有没有适合两个人约会、人均80以内、不要火锅的餐厅？",
  "conversationId": "optional-conversation-id",
  "city": "北京",
  "longitude": 116.3103,
  "latitude": 39.9836,
  "scene": "recommendation"
}
```

前端处理：

1. 普通聊天使用 Axios。
2. 流式聊天因当前后端为 POST SSE，优先使用 `fetch` + `ReadableStream` 解析事件流。
3. 渲染 `meta`、`chunk`、`done`、`error` 事件。
4. 展示 answer、conversationId、traceId。
5. 可折叠展示 plan、toolTrace、retrievalHits，方便演示 AI 链路。
6. AI 不应在前端补写价格、库存、距离等动态事实。
7. AI 对话页应支持“普通问答”和“流式问答”切换，流式模式必须逐段追加 `chunk`。

## 7. 接口响应规范

非 SSE 接口统一返回：

```ts
export type ApiResult<T> = {
  success: boolean;
  code: string;
  message: string;
  data: T;
  traceId: string;
  timestamp: number;
};
```

前端必须统一处理：

1. `success=true`：使用 `data`。
2. `success=false`：展示 `message`，必要时展示 `traceId`。
3. HTTP 401 或业务码 `UNAUTHORIZED`：跳转登录页。
4. HTTP 429 或业务码 `AI_RATE_LIMITED`：提示请求过于频繁。
5. 其他异常：进入通用错误提示，不吞掉 traceId。

## 9. 状态管理规范

### 9.1 TanStack Query 管理的数据

以下属于后端服务端数据，应使用 TanStack Query：

1. 商户详情。
2. 商户列表。
3. 商户分类。
4. 优惠券列表。
5. AI 普通问答结果。
6. 会话清理结果。

### 9.2 Zustand 管理的数据

以下属于前端全局状态，应使用 Zustand：

1. 登录 token。
2. 当前用户信息。
3. 当前定位。
4. 当前 AI conversationId。
5. 全局弹窗或 toast 状态。

### 9.3 不应放入全局状态的数据

1. 单个页面局部 loading。
2. 表单临时输入值。
3. 单个卡片 hover 或展开状态。
4. 可以通过接口重新获取的列表数据。

## 10. 金额、时间和状态展示规范

### 10.1 金额

后端金额字段以“分”为主，例如：

```text
avgPrice
payValue
actualValue
```

前端展示时必须转换为“元”：

```ts
function formatFen(value?: number | null): string {
  if (value == null) return "价格待确认";
  return `¥${(value / 100).toFixed(2)}`;
}
```

### 10.2 秒杀时间

秒杀券包含：

```text
beginTime
endTime
```

前端根据当前时间展示：

1. 未开始。
2. 进行中。
3. 已结束。
4. 已抢光。
5. 已抢过。

最终状态以后端返回为准，前端只做展示和按钮禁用。

### 10.3 traceId

所有错误提示建议附带 traceId：

```text
请求失败，请稍后重试
traceId: xxxxx
```

这样方便后端排查日志。

## 11. AI 流式响应规范

当前后端接口：

```text
POST /ai/agent/chat/stream
Content-Type: text/event-stream
```

后端事件类型：

| 事件 | 含义 |
| --- | --- |
| `meta` | 会话 ID、traceId、意图等元信息。 |
| `chunk` | 一段回答文本。 |
| `done` | 回答结束。 |
| `error` | 流式回答失败。 |

前端要求：

1. 发送消息后立即显示用户消息。
2. 接收 `chunk` 时逐段追加 AI 消息。
3. 收到 `done` 后结束 loading。
4. 收到 `error` 后展示错误卡片和 traceId。
5. 断流时允许用户重试，但不要重复提交不可幂等业务请求。

## 12. 权限与登录态

当前后端登录接口返回 token，但部分 AI 接口使用 `X-User-Id` / `X-Principal-Key` 请求头进行演示级身份标识。前端开发时应按以下策略封装：

1. 登录成功后保存 token、userId、nickName。
2. 普通业务接口预留 `Authorization` 或 `X-Token` 注入位置，等待后端鉴权过滤器完善。
3. AI 演示接口请求头携带 `X-User-Id`。
4. 未登录访问秒杀、AI 会话等页面时，引导到登录页。
5. 普通 Axios 请求和 POST SSE `fetch` 请求都必须复用登录态请求头策略，至少在已有登录用户时携带 `X-User-Id`。

不得在前端硬编码固定 userId 作为正式逻辑。演示数据必须明确标注为本地演示用途。

## 13. 错误处理和空状态

必须覆盖以下状态：

1. 商户不存在。
2. 商户列表为空。
3. 优惠券为空。
4. 秒杀库存不足。
5. 重复抢券。
6. 秒杀未开始或已结束。
7. AI 限流。
8. AI 服务不可用。
9. RAG 无命中。
10. 网络异常。

错误提示要让用户知道下一步能做什么，例如重新加载、返回首页、换个问题、重新登录。

## 14. 前后端联调验收路径

后续前端工程完成后，至少跑通以下路径：

1. 登录：输入手机号，成功进入首页。
2. 商户：从首页进入商户列表，再进入商户详情。
3. 优惠券：在商户详情页看到普通券和秒杀券。
4. 秒杀：点击秒杀券，成功时进入结果页，失败时展示明确原因。
5. AI 普通问答：发送客服问题，展示 answer、traceId、retrievalHits。
6. AI 推荐问答：发送推荐需求，展示 toolTrace 和推荐相关回答。
7. AI 流式问答：逐段渲染 chunk，最终收到 done。
8. 会话清理：清理 conversationId 后新开对话。
9. 异常：断网、未登录、限流、后端错误时均有友好提示。

## 16. 后续推荐开发顺序

建议分四轮开发：

1. 第一轮：已创建 React + TypeScript + Vite 工程骨架，完成路由、样式 token、Axios 封装、登录页、首页、商户/优惠券/秒杀/AI 页面骨架，并完成 Review 修复。
2. 第二轮：完成真实接口联调，重点验证登录、商户列表、商户详情、优惠券列表、秒杀失败提示和 AI 普通/流式问答。
3. 第三轮：补充 toolTrace、retrievalHits、AI 推荐解释、SSE 边界状态和会话管理体验。
4. 第四轮：补充错误状态、空状态、移动端适配、演示脚本和端到端联调。

每轮开发完成后必须更新本文档和 `AGENTS.md`，记录已完成页面、接口联调结果、剩余风险和下一步任务。
